# Testing Guideline

## 目的

テストは「実装の確認」ではなく「想定の確立」のために書く。

- このシステムは何を約束しているか
- その約束が壊れたとき、テストが教えてくれるか

実装の中身（どの関数を呼んでいるか、内部の分岐がどうなっているか）には依存しない。
入力と出力の関係、つまり **振る舞いの契約** をテストする。

## 3A パターン

すべてのテストは Arrange / Act / Assert の 3 段構成で書く。

```rust
#[test]
fn 検索はタグにもマッチする() {
    // Arrange: テストに必要な状態を組み立てる
    let snippets = vec![snippet_with_tags("deploy", &["aws", "prod"])];

    // Act: テスト対象の操作を 1 つだけ実行する
    let results = filter_snippets(&snippets, "aws");

    // Assert: 期待する結果を検証する
    assert_eq!(results.len(), 1);
    assert_eq!(results[0].name, "deploy");
}
```

### 守ること

- **Arrange は明示的に。** ヘルパーで組み立てるのはよいが、何を作ったか読めば分かるようにする
- **Act は 1 つ。** 1 テスト 1 操作。複数の操作を連鎖させない
- **Assert は結果だけ。** 戻り値・ファイルの中身・エラーの種類を検証する。内部状態は見ない

### 避けること

- Arrange の中で別のテスト対象を呼ぶ（テスト同士が依存する）
- Act で複数の操作をして「最終状態」だけ見る（どこで壊れたか分からない）
- Assert で `.unwrap()` して「パニックしなければ OK」とする（何を期待しているか不明）

## テストの分類

### 1. 契約テスト（純粋関数）

副作用のない関数の入出力を検証する。最も書きやすく、最も信頼できる。

対象:
- プレースホルダのパース (`parse_placeholders`)
- YAML スカラーのエスケープ (`yaml_scalar`)
- 検索のフィルタリングロジック

### 2. 往復テスト（シリアライズ）

「書いて読み戻したら同じものが得られる」ことを検証する。
データファイルのフォーマットが壊れていないことの保証。

対象:
- Snippet の YAML roundtrip
- `append_snippet` で書いたものを `load_snippets` で読める
- 最小構成（name + description だけ）でも往復する

### 3. ファイル操作テスト

一時ファイルを使って、ファイルシステムとの境界を検証する。
`SNIPPET_FILE` 環境変数で制御する。

対象:
- 空ファイル / 存在しないファイルからの読み込み
- 追記がファイル内容を壊さないこと
- 既存コメントが `append_snippet` で消えないこと

### テスト対象外

- 対話的 I/O（`dialoguer` のプロンプト）→ ロジックを分離してロジック側をテスト
- 外部プロセス実行（`sh -c`）→ 統合テストの範囲だが v1 では書かない
- `process::exit` を呼ぶパス → テストプロセスが死ぬので避ける

## テストヘルパーの方針

- `fn make_snippet(name, desc)` — 最小構成の Snippet を作る
- `fn make_snippet_full(...)` — 全フィールド指定
- `fn with_temp_file(content, f)` — 一時ファイルを作って `SNIPPET_FILE` をセット、テスト後に掃除

ヘルパーは `tests/` やテストモジュール内に閉じる。プロダクションコードに置かない。

## 命名

テスト名は「何が・どうなる」を日本語で書く。実装の関数名を繰り返さない。

```rust
// Good: 振る舞いが分かる
fn 検索クエリなしで全件返る()
fn デフォルト値つきプレースホルダを展開できる()
fn 追記しても既存エントリが壊れない()

// Bad: 実装をなぞっているだけ
fn test_search_none()
fn test_parse_placeholders_with_defaults()
```

## テスタビリティのために

現在の実装で直接テストしにくい部分は、ロジックを純粋関数に切り出す。

例: `commands::search` は stdout に書いて `Result<()>` を返すが、
フィルタリングロジックだけ `fn filter_snippets(snippets, query) -> Vec<&Snippet>` に切り出せば
I/O なしでテストできる。

この切り出しは「テストのためのリファクタリング」ではなく「関心の分離」。
I/O と判断ロジックが混ざっている関数は、テスト以前に読みにくい。
